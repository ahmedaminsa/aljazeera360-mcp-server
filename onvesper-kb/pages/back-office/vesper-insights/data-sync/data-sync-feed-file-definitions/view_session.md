# View\_Session

The view\_session provides a comprehensive overview of viewership sessions, which are understood as continuous periods of time during which a user is engaged in watching content.

A user can either be:

* a customer (someone who has made purchases, whether a single transaction like a PPV or a subscription, regardless of its current status)
* a guest-user (someone who gained access to content without making any purchase)

The view\_session contains the following information:

* **WHO:** Identifies the customer who watched the specific content
* **WHEN:** Indicates the date and time when the customer watched the content
* **WHAT:** Describes the content and the viewership session that the customer engaged in, including its content title, video type, duration in seconds, and other relevant details
* **HOW:** Specifies the device used, country of viewing, or other means utilized by the customer to watch the content or engage in the viewership session

### How a View Session is Computed and Stored

1. When a user starts viewing content, a VOD\_WATCHING\_START or LIVE\_WATCHING\_START event is sent, and at that time, a view\_session is created.
2. A view\_session is ended (aka closed), when a VOD\_WATCHING\_END or LIVE\_WATCHING\_END is sent. At that time, a view\_session ends, and it’s endedAt field set, along with computing a few other fields such as duration\_sec, etc.
3. If for some reason a subsequent VOD/LIVE\_WATCHING\_START event is received with the same computed session\_id and without first receiving a VOD/LIVE\_WATCHING\_END event, then we know the user is still watching the same piece of content and we simply continue their existing view session.
   * We make a note of the datetime of the most recent VOD/LIVE\_WATCHING\_START event received and use that if we never receive an end event and need to later know what date to use to end such a view session.
4. If a new VOD/LIVE\_WATCHING\_START event is received with the same computed session\_id within 6 minutes of the same view session that previously ended (ie: we previously received a VOD/LIVE\_WATCHING\_END event), then we know the user is still watching the same piece of content and we extend the previous watched view session and the view session remains ongoing.
   * However, if we receive a VOD/LIVE\_WATCHING\_START event after 6 minutes with the same computed session\_id of a previous view session, then a new view session is created, even though the same computed session\_id was received.
5. If a VOD/LIVE\_WATCHING\_END event is never received, then after 24 hours a job is run to terminate any view session open for more than 24 hours, and we set the end date of that view session equal to the last VOD/LIVE\_WATCHING\_START event received + 6 minutes.
   1. If we later receive a VOD/LIVE\_WATCHING\_END event for that same view session after 24 hours, which is after we closed the view session, we then update that view session end date to be the new end date that we just received.
6. If a user pauses their view session, no VOD/LIVE\_WATCHING\_END event is sent until the configured system level time has elapsed where no heartbeat has been received. Otherwise, the view session remains open until they start watching again and leave the session, this will result in a VOD/LIVE\_WATCHING\_END event being sent and we end the view session. But if for some other reason no VOD/LIVE\_WATCHING\_END event has been received for more than 24 hours, then we will end the view session using an end datetime of the last start event we received + 6 minutes.
7. Ad breaks, or moving the player forward or reverse will have no impact on the computation of the starting or ending of a view session. Both ads and moving the player head do not send events and constitutes still watching the content for the purposes of a view session.

### Computing the view's session\_id

Each view created in view\_session, has a unique computed session\_id. It is an md5 hashed concatenation of realm, cid, video\_type, content\_id, and watch\_device. For those technical audience readers, it looks like this:

```
// ,md5(
    coalesce(realm::varchar,'') || '|' ||
    coalesce(cid::varchar,'') || '|' ||
    coalesce(video_type::varchar,'') ||
    coalesce(content_id::varchar,'') ||
    coalesce(watch_device::varchar,'')
) as session_id
```

The reason we cannot rely on cid alone is because of multi-view players. Users can watch up to 4 different pieces of content at the same time. In order to allow this without hitting the user’s concurrency limit, the cid is shared between each viewed content in the multi-view player. In this case, we have 4 different view session running with different content, but with the same cid.

### FAQ

This section attempts to answer the most frequent questions about data you find in this view\_session data sync feed

1. If a user watches content through the browser on a mobile device what value will we expect to see for the watch\_device field:
   * The view will appear as watch\_device = BROWSER. Unfortunately there is no way to tell today from the platform. It’s not that we don’t surface it, we don’t have the differentiating data to surface it from the apps into analytics at this time.
   * From this, the benefit you gain is that if you see a view\_session with watch\_device that is not BROWSER, such as IOS\_PHONE, it means that the user watched using the app on their phone and not the browser on the phone. Which is important metric business users tend to want to track/see.
2. There are a lot of records where progress is significantly bigger than content\_duration. This also then causes the video\_completion\_rate to be greater than 100% (by a lot).
   * This is what is currently being received. the progress field is reported by the client apps. Some client apps are implemented by 3rd parties (not ES), if strange values are present, please contact the the 3rd party to correct, or ES TAM to assist with communications. It is usually an issue with the implementation.
3. There are a lot of records where duration\_sec is much greater than the content\_duration.
   * This is what is currently being received. This is based on the client apps sending heartbeats. Some client apps are implemented by 3rd parties (not ES). if strange values are present, please contact the the 3rd party to correct, or ES TAM to assist with communications. It is usually an issue with the heart beat implementation being reported by the apps. For instance, if the user has paused a video but keeps sending a heart beat, that will keep the session live beyond the length of the video itself and result in a very large value for duration\_sec. Typically, Heartbeats would not be sent during pausing a video.
